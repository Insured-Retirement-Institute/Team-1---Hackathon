# Servicing Agent Change Request API — Steps 4 & 5

## Overview

After the Receiving Firm completes due diligence on the policies returned from the Policy Inquiry (Steps 1–3), it submits a Servicing Agent Change Request through DTCC to the insurance carrier for each in-scope client/carrier combination. The carrier validates the request against licensing, appointments, suitability, and policy rules, then responds with an approval, rejection, or pending status per policy.

This is a **single client, single carrier** request. If a client has policies across multiple carriers, separate requests are submitted for each carrier.

---

## Step 4 — Servicing Agent Change Request

Submitted by the Receiving Broker/Dealer through DTCC to the Insurance Carrier.

**DTCC Status:** `CARRIER_VALIDATION_PENDING`

### Payload

```json
{
  "servicingAgentChangeRequest": {
    "requestId": null,
    "requestingFirm": {
      "firmName": null,
      "firmId": null,
      "servicingAgent": {
        "agentName": null,
        "npn": null,
        "carrierSpecificId": null
      }
    },
    "carrier": {
      "carrierName": null,
      "carrierId": null
    },
    "client": {
      "clientName": null,
      "brokerageAccountNumber": null,
      "policyNumbers": []
    }
  }
}
```

### Field Reference

| Field | Type | Description |
|-------|------|-------------|
| `requestId` | string (ULID) | Universally Unique Lexicographically Sortable Identifier generated by the requesting firm, used for subsequent status polling |
| `requestingFirm.firmName` | string | Name of the receiving broker/dealer |
| `requestingFirm.firmId` | string | DTCC firm identifier |
| `requestingFirm.servicingAgent.agentName` | string | Name of the new servicing agent |
| `requestingFirm.servicingAgent.npn` | string | National Producer Number of the new agent |
| `requestingFirm.servicingAgent.carrierSpecificId` | string | Carrier-specific identifier for the producer, if applicable |
| `carrier.carrierName` | string | Name of the insurance carrier |
| `carrier.carrierId` | string | Carrier identifier |
| `client.clientName` | string | Full name of the client |
| `client.brokerageAccountNumber` | string | Optional. Brokerage account number that will hold the policy upon change |
| `client.policyNumbers` | string[] | List of policy numbers issued by this carrier for this client |

### Notes

- The `requestId` is a [ULID](https://github.com/ulid/spec) generated by the requesting firm. ULIDs are lexicographically sortable and encode a timestamp, making them useful for both correlation and chronological ordering of requests.
- All policy numbers in a single request must belong to the same carrier.
- One client per request.

## Step 5 — Servicing Agent Change Response

Returned by the Insurance Carrier through DTCC to the Receiving Firm.

**DTCC Status:** `CARRIER_APPROVED`, `CARRIER_REJECTED`, or `CARRIER_PENDING`

### Payload

```json
{
  "servicingAgentChangeResponse": {
    "requestId": null,
    "carrier": {
      "carrierName": null,
      "carrierId": null
    },
    "policies": [
      {
        "policyNumber": null,
        "status": null,
        "effectiveDate": null,
        "errors": []
      }
    ]
  }
}
```

### Field Reference

| Field | Type | Description |
|-------|------|-------------|
| `requestId` | string (ULID) | Echoed request identifier for correlation and status polling |
| `carrier.carrierName` | string | Carrier name |
| `carrier.carrierId` | string | Carrier identifier |
| `policies` | array | List of policy-level results |

#### Policy Object

| Field | Type | Description |
|-------|------|-------------|
| `policyNumber` | string | Policy number |
| `status` | enum | Current status of the servicing agent change for this policy (see enums) |
| `effectiveDate` | string (ISO 8601) | Effective date of the agent change, if approved. Null if rejected or pending. |
| `errors` | array | Any errors associated with this policy (see error catalog) |

### Notes

- Each policy in the request receives its own status — some may be approved while others are rejected or pending within the same response.
- The `requestId` can be used to poll the Status API for updates on policies with a `pendingAppointment` status.
- `effectiveDate` is only populated when `status` is `approved`.

---

## Enums

### status

| Value | Description |
|-------|-------------|
| `approved` | servicing agent change validated and accepted by the carrier |
| `rejected` | servicing agent change rejected by the carrier (see errors for reason) |
| `pendingAppointment` | Producer appointment has been initiated by the carrier (JIT appointment); recheck via Status API |

---

## Error Catalog

All errors are returned in the `errors` array within each policy object. Each error includes a severity indicating whether the issue is a hard stop (request cannot proceed) or a soft stop (request is in progress, pending resolution).

### Error Object

```json
{
  "errorCode": null,
  "severity": null,
  "message": null
}
```

| Field | Type | Description |
|-------|------|-------------|
| `errorCode` | string | Machine-readable error identifier |
| `severity` | enum | `hard` (request cannot proceed) or `soft` (in progress, pending resolution) |
| `message` | string | Human-readable description of the error |

### Error Codes

| Error Code | Severity | Message | Description |
|------------|----------|---------|-------------|
| `competingChangeRequest` | hard | Policy is subject to a competing change request | Another servicing agent change is already in progress for this policy |
| `pendingDeathClaim` | hard | Policy is subject of a pending death claim | A death claim has been filed against this policy |
| `proprietaryProduct` | hard | Policy is a proprietary product and not available for an agent change | The product is proprietary to the current distributor |
| `notAppointedJit` | soft | Producer is not appointed — appointment initiated by carrier | The carrier has initiated a just-in-time appointment for the producer |
| `notAppointedResubmit` | hard | Producer is not appointed — resubmit after appointment | The producer must obtain an appointment before the request can be processed |
| `notLicensed` | hard | Producer is not licensed | The producer does not hold the required license |
| `carrierSpecific` | hard | Carrier specific error — see description | A carrier-defined error; details provided in the message field |
| `pendingWithdrawalSurrender` | hard | Policy is subject to a pending full withdrawal/surrender | A full withdrawal or surrender is in progress on this policy |
| `policyNotInForce` | hard | Policy is no longer in force | The policy has terminated and is no longer active |
| `noSellingServicingAgreement` | hard | Product is not covered by selling or servicing agreement between carrier and new distributor | The requesting firm does not have an active selling or servicing agreement with the carrier for this product |

## Status Polling

The `requestId` returned in both the request and response can be used to poll the Status API for updates on pending items. This is particularly useful for policies in `pendingAppointment` status where the carrier has initiated a JIT appointment.

```
GET /status/{requestId}
```

The Status API returns the current state of all policies within the original request, using the same response structure as the Servicing Agent Change Response.

---

## Sequence Context

This request/response pair covers Steps 4 and 5 of the broader Servicing Agent Change Process. Upon receiving an `approved` status, the Receiving Firm initiates a formal transfer request (Step 6). Policies with `rejected` status require no further action. Policies with `pendingAppointment` status should be monitored via the Status API until resolved.
